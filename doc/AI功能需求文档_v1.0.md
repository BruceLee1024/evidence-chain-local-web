# 证据链管理系统 — AI 功能需求文档 v1.0

> 编制日期：2026-06-20
> 版本：v1.0
> 基于：需求文档_v1.0.md、现有系统能力分析

---

## 一、背景与目标

### 1.1 为什么需要 AI

当前系统已实现证据链管理的核心闭环（录入 → 组卷 → 导出），但在以下环节仍依赖大量人工操作：

| 环节 | 现状 | 人工成本 |
|------|------|---------|
| 证据录入 | 对着扫描件逐字段手敲 | 每条 3-5 分钟 |
| 资料查找 | SQL LIKE 关键词匹配，搜不到就是找不到 | 翻箱倒柜 |
| 证据匹配 | 规则引擎只能做「同词匹配」，不同叫法就漏掉 | 人工补漏 |
| 组卷检查 | 靠经验判断证据是否齐全 | 甲方指出才补 |

**目标**：在不改变现有核心流程的前提下，用 AI 增强录入效率、搜索精准度和匹配覆盖率。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **AI 建议 + 人工确认** | 所有 AI 输出都是辅助建议，最终决定权在用户 |
| **渐进增强** | AI 作为现有功能的增强层，不影响已稳定的 CRUD + 匹配流程 |
| **数据安全可配** | 支持云端 AI API 和本地模型（Ollama），敏感数据可不出本机 |
| **成本可控** | AI 调用只在用户主动触发时发生（上传文件、搜索、生成匹配），不做后台全量扫描 |

### 1.3 适用范围

本文档涵盖 AI 增强功能的三个阶段规划。Phase 1 为 v1.0 AI 版本的核心交付范围。

## 二、现有系统 AI 可用资产分析

| 资产 | 位置 | AI 可用度 |
|------|------|----------|
| 4 类证据的结构化 payload | `evidence.payload_json`（JSON 字段） | ⭐⭐⭐ 高 — 可直接供 AI 推理 |
| 上传文件（照片/PDF/扫描件） | `data/` 目录，经 `files` + `evidence_files` 表管理 | ⭐⭐⭐ 高 — OCR 金矿，内容未被解析 |
| 结算项 + 匹配关系 | `settlement_items` + `settlement_links` 表 | ⭐⭐⭐ 高 — 有置信度 + 人工校对记录可供 AI 学习 |
| 部位结构（楼栋→楼层→部位） | `locations` 表 + `normalized_location` | ⭐⭐ 中 — 可辅助 AI 推理层级关系 |
| 全文搜索 | SQL `LIKE` 查询 `title/code/location/payload_json` | ⭐⭐ 中 — 可升级为语义搜索 |

## 三、AI 功能规划

### Phase 1（v1.0 AI 版 · 核心交付）

---

#### 功能 1：智能 OCR 录入

**优先级**：🔴 P0
**解决痛点**：录入签证单/验收单时需对着扫描件逐字段手敲

**功能描述**：

用户在证据表单中上传扫描件/照片后，点击「AI 识别」按钮，系统调用 AI Vision 模型识别文件中的关键字段，自动填入表单。用户确认或修正后保存。

**覆盖证据类型**：

| 证据类型 | AI 识别目标字段 | 输入文件类型 |
|---------|---------------|------------|
| 签证变更单 | 变更编号、变更类型、变更原因、增减金额、签证日期、施工单位签字人、监理签字人、甲方签字人 | 扫描件/照片 |
| 隐蔽工程记录 | 部位、工序、验收日期、验收结论 | 验收单扫描件/照片 |
| 材料进场记录 | 材料名称、规格型号、单位、数量、品牌厂家、供应商、进场日期 | 合格证/检测报告/验收照片 |
| 月度计量记录 | 计量月份、本期完成产值、累计完成产值、甲方签字人、确认日期 | 确认单扫描件 |

**交互流程**：

```
用户进入证据表单
  → 上传文件（扫描件/照片）
  → 点击「🤖 AI 识别填写」
  → 服务端调用 AI Vision API 识别
  → 返回结构化字段预填表单
  → 用户确认/修正 → 保存
```

**技术方案**：

```
POST /api/ai/ocr/extract

Content-Type: multipart/form-data
  - file: 扫描件/照片
  - evidenceType: 'variation' | 'hidden' | 'material' | 'monthly'

响应：
{
  fields: {
    code: "2026-023",
    changeType: "现场签证",
    reason: "基坑超挖增加混凝土量",
    amount: 120000,
    signDate: "2026-05-15",
    ...
  },
  confidence: 0.87,       // 整体置信度
  fieldConfidence: {       // 逐字段置信度
    code: 0.95,
    amount: 0.92,
    reason: 0.78,
    ...
  }
}
```

**容错设计**：
- 每个字段标注置信度，低置信度（< 0.7）字段高亮提示用户重点检查
- 识别失败时降级为手动录入，不阻塞流程
- 支持用户反馈「识别不准」以持续优化 prompt

**验收标准**：
1. ✅ 4 类证据表单均有「AI 识别填写」按钮
2. ✅ 上传文件后点击按钮，10 秒内返回识别结果
3. ✅ 识别结果正确预填到对应表单字段
4. ✅ 低置信度字段有明显视觉提示
5. ✅ 识别失败时有友好降级提示

#### 功能 2：语义搜索

**优先级**：🔴 P0
**解决痛点**：「3号楼地下室那批钢筋」搜不到「3#楼地下室底板」关联的证据

**功能描述**：

升级全局搜索，在现有关键词匹配基础上增加语义搜索层。用户输入自然语言查询（如「地下室防水的签证单」「王经理签过字的计量单」），系统用 Embedding 向量做语义相似度匹配，按相关度排序结果。

**搜索增强矩阵**：

| 搜索层 | 技术 | 覆盖场景 |
|--------|------|---------|
| 第一层：精确匹配 | SQL LIKE / 正则 | 编号、精确日期、完整名称 |
| 第二层：语义匹配 | Embedding 向量 + 余弦相似度 | 同义词、不同表述、自然语言 |
| 第三层：结构化过滤 | 现有 type/date 筛选 | 缩小范围 |

**交互变化**（现有搜索页升级）：

```
┌─────────────────────────────────────────────┐
│  🔍  [   地下室底板相关的所有证据           ] │
│       [ 全部类型 ▼]  [ 最近6个月 ▼]         │
│                                              │
│  💡 AI 建议：你是不是想找...                   │
│  ─────────────────────────────────────────── │
│  结果（按相关度排序）：                         │
│  ████████████ 92% 隐蔽工程 · 地下室底板钢筋验收  │
│  ██████████░░ 85% 签证单 · #2026-018 防水变更   │
│  ████████░░░░ 72% 材料进场 · C35混凝土 120m³    │
│  ██████░░░░░░ 58% 月度计量 · 2026-06           │
└─────────────────────────────────────────────┘
```

**技术方案**：

```
方案：文本向量化 + SQLite 存储向量

索引构建（后台/安装时执行一次）：
  → 遍历所有 evidence 记录
  → 拼接「type + title + location + payload 摘要」为搜索文本
  → 调用 Embedding API 生成向量
  → 存入 evidence_embeddings 表

搜索时：
  GET /api/search/semantic?q=地下室底板钢筋
  → 对查询文本生成向量
  → 与 evidence_embeddings 表做余弦相似度计算
  → 返回 topK 结果（含相似度分数）
```

**新增数据表**：

```sql
CREATE TABLE IF NOT EXISTS evidence_embeddings (
  evidence_id TEXT PRIMARY KEY REFERENCES evidence(id) ON DELETE CASCADE,
  embedding BLOB NOT NULL,        -- 向量二进制存储
  search_text TEXT NOT NULL,      -- 原始搜索文本（用于重建索引）
  model TEXT NOT NULL,            -- Embedding 模型标识
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

**重建索引策略**：
- 新增/修改 evidence 时自动更新对应向量
- 提供「重建语义索引」按钮供手动触发全量重建
- Embedding 模型切换时需重建

**验收标准**：
1. ✅ 搜索输入框支持自然语言（非单一关键词）
2. ✅ 结果按语义相关度排序，显示相关度百分比
3. ✅ 同义词/不同表述能匹配到正确结果（如「三号楼」↔「3#楼」）
4. ✅ 与现有类型/日期筛选组合使用
5. ✅ 新建证据后语义索引自动更新

#### 功能 3：AI 增强匹配

**优先级**：🟡 P1
**解决痛点**：规则引擎只能做字符串精确/包含匹配，不同叫法的证据漏掉

**功能描述**：

在现有 `matching.js` 规则引擎的基础上增加 AI 语义匹配层。对规则引擎判定为 `rejected`（confidence=0）的「证据-结算项」配对，用 AI 做二次语义判断，捞回被误拒的匹配。

**匹配流程（增强后）**：

```
结算项 + 证据
  │
  ├── 第一层：规则引擎（现有 matching.js）
  │   ├── 精确匹配 → auto（confidence 92）
  │   ├── 模糊匹配 → candidate（confidence 68）
  │   └── 不匹配 → rejected（confidence 0）→ 进入第二层
  │
  └── 第二层：AI 语义匹配（新增）
      ├── AI 判断相关 → candidate（confidence AI打分）
      └── AI 判断不相关 → 保持 rejected
```

**AI prompt 设计**：

```
你是一个建筑工程结算证据审核助手。判断以下证据是否可能支撑给定的结算项。

结算项名称：{itemName}
结算项部位：{itemLocation}
结算项时段：{startDate} 至 {endDate}

证据类型：{evidenceType}
证据标题：{evidenceTitle}
证据部位：{evidenceLocation}
证据日期：{evidenceDate}
证据摘要：{summary}

请判断这个证据是否可能支撑该结算项。
注意：同一实体的不同写法（如"3#楼"和"三号楼"）应视为匹配。
不同工序但同一部位和时间段的证据也可能相关。

返回 JSON：
{
  "relevant": true/false,
  "confidence": 0-100,
  "reason": "简短说明"
}
```

**性能控制**：
- 只在规则引擎 rejection 上做 AI 调用，避免全量调用
- 单次组卷 AI 调用上限：50 次（超出用规则引擎结果）
- 缓存同一次组卷中对同一证据的 AI 判断结果

**验收标准**：
1. ✅ 组卷匹配结果中出现 `matchKind: 'AI语义匹配'` 的 evidence
2. ✅ AI 匹配到的证据与人工判断一致率 > 80%
3. ✅ 一个结算项的 AI 增强匹配在 30 秒内完成
4. ✅ 用户可手动确认/移除 AI 建议的匹配

#### 功能 4：证据完整性检查

**优先级**：🟡 P1
**解决痛点**：组卷时不知道证据是否齐全，结算时被甲方指出才补

**功能描述**：

导入结算项并完成匹配后，AI 分析每个结算项的证据覆盖情况，标记缺失的证据类型并给出建议。

**检查维度**：

| 结算项类型（AI 推断） | 期望证据 | 检查逻辑 |
|---------------------|---------|---------|
| 混凝土工程 | 隐蔽验收记录 + 材料进场记录 + 月度计量 | 关键词含「混凝土」→ 至少各 1 条 |
| 钢筋工程 | 隐蔽验收记录 + 材料进场记录（钢筋） | 关键词含「钢筋」→ 至少各 1 条 |
| 防水工程 | 隐蔽验收记录 | 关键词含「防水」→ 至少 1 条 |
| 签证变更 | 签证变更单 | 关键词含「变更」→ 至少 1 条 |
| 通用 | 月度计量确认单 | 所有结算项建议关联 1 条 |

**输出示例**：

```
结算项：3#楼地下室底板混凝土  状态：⚠️ 证据不完整

  ✅ 隐蔽工程验收记录（1条）  — 地下室底板钢筋验收 2026-05-15
  ✅ 材料进场记录（1条）      — C35混凝土 120m³ 2026-05-10
  ⚠️ 缺少月度计量确认单      — 建议补传 2026-05 或 2026-06 的计量单

结算项：3#楼外墙防水         状态：🚨 缺少关键证据

  ❌ 缺少隐蔽工程验收记录    — 防水工程必须提供验收记录
  ❌ 缺少材料进场记录        — 建议补充防水材料进场证明
  ⚠️ 缺少月度计量确认单      — 建议补传对应时段计量单
```

**交互位置**：结算组卷页「匹配校对」面板中，每个结算项旁显示完整性状态图标。

**验收标准**：
1. ✅ 组卷完成后每个结算项有完整性状态（✅完整 / ⚠️不完整 / 🚨严重缺失）
2. ✅ 点击状态图标展开 AI 建议的缺失证据类型
3. ✅ AI 建议准确率 > 70%（基于结算项名称推断需要的证据类型）
4. ✅ 用户可忽略 AI 建议（不阻塞流程）

#### 功能 5：自然语言快速录入

**优先级**：🟡 P1（Phase 2）
**解决痛点**：表单字段多，希望能「一句话录入」

**功能描述**：

在证据表单顶部增加「快速录入」输入框。用户用自然语言描述 → AI 解析为结构化字段 → 预填表单。

**示例**：

```
输入：「6月15号进了120吨首钢的HRB400螺纹钢直径25 张三收的货」
  → 自动填入：
    进场日期: 2026-06-15
    材料名称: HRB400 螺纹钢
    规格型号: Φ25
    进场数量: 120
    单位: 吨
    品牌厂家: 首钢
    收货人: 张三
```

**技术方案**：

```
POST /api/ai/nlu/parse

{
  text: "用户的自然语言描述",
  evidenceType: "material" | "variation" | "hidden" | "monthly"
}

响应：
{
  fields: { ... },          // 结构化字段
  confidence: { ... },      // 逐字段置信度
  missingRequired: [...]    // 必填但未识别到的字段
}
```

**覆盖场景**：4 类证据均支持，特别适合材料进场（字段多但规律性强）和签证单（描述性信息多）。

**验收标准**：
1. ✅ 4 类证据表单均有「快速录入」入口
2. ✅ 自然语言描述在 5 秒内返回结构化结果
3. ✅ 常见施工用语识别准确率 > 85%
4. ✅ 必填字段未被识别时有提示

#### 功能 6：结算报告 AI 生成

**优先级**：🟢 P2（Phase 2）
**解决痛点**：导出 ZIP 只有文件，缺少汇总说明

**功能描述**：

在结算组卷导出环节增加「生成结算说明报告」选项。AI 基于组卷数据生成一份正式的结算证据说明报告（Word/PDF），包含项目概况、结算项清单、证据摘要、缺失项说明。

**报告结构**：

```
XX项目 结算证据说明报告
生成日期：2026-06-20

一、项目概况
  - 项目名称、结算组卷批次、导出时间

二、结算项清单（共 N 项）
  | 序号 | 结算项 | 已匹配证据数 | 状态 |
  
三、证据详情
  3.1 地下室底板混凝土
    - 支撑证据 1：隐蔽工程验收记录 · 底板钢筋验收 · 2026-05-15
    - 支撑证据 2：材料进场记录 · C35混凝土 120m³ · 2026-05-10
    - 缺失项：[无]
  
  3.2 外墙防水
    - 支撑证据：[无]
    - 缺失项：隐蔽工程验收记录、材料进场记录
    - 说明：该结算项目前无支撑证据，建议补充后重新组卷

四、汇总统计
  - 完整结算项：X 项
  - 部分完整：Y 项
  - 缺失证据：Z 项
```

**验收标准**：
1. ✅ 组卷预览页面增加「生成报告」按钮
2. ✅ 报告为 Word（.docx）格式，可编辑
3. ✅ 报告内容准确反映组卷数据
4. ✅ 缺失项有明确标注和建议

## 四、技术架构

### 4.1 新增模块结构

```
server/
├── ai/                          # 新增 AI 模块
│   ├── index.js                 # AI 路由注册
│   ├── ocr.js                   # 功能 1：OCR 图片识别
│   ├── embedding.js             # 功能 2：文本向量化 + 语义搜索
│   ├── matcher.js               # 功能 3：AI 增强匹配
│   ├── checker.js               # 功能 4：证据完整性检查
│   ├── nlu.js                   # 功能 5：自然语言解析
│   ├── report.js                # 功能 6：结算报告生成
│   └── provider.js              # AI Provider 抽象层（统一 API 调用）
├── lib/
│   ├── matching.js              # 现有规则引擎（功能 3 会调用 AI 作为补充）
│   └── ...
└── index.js                     # 现有 Express 路由
```

### 4.2 AI Provider 抽象层

```
server/ai/provider.js

支持三种 Provider，通过配置切换：

1. DeepSeek（默认推荐）
   - 性价比高，中文能力强
   - API: https://api.deepseek.com/v1

2. OpenAI / Claude
   - 视觉能力更强（适合 OCR）
   - API: https://api.openai.com/v1

3. Ollama（本地）
   - 数据不出本机，适合对隐私要求极高的场景
   - 需用户自行部署 Ollama 服务
   - 注意：本地模型视觉能力有限

配置方式（config.js 或环境变量）：
  AI_PROVIDER=deepseek|openai|ollama
  AI_API_KEY=sk-xxx
  AI_BASE_URL=https://api.deepseek.com/v1
  AI_MODEL=deepseek-chat
  AI_VISION_MODEL=deepseek-vl2
  AI_EMBEDDING_MODEL=text-embedding-3-small
```

### 4.3 API 路由规划

```
# OCR 识别

  → 上传文件 → 返回结构化字段

  → 返回语义排序的搜索结果

# AI 增强匹配（在组卷时自动触发，也可手动重新匹配）

  → 对 unmatched/partial 的结算项重新做 AI 匹配
```

## 五、非功能需求

### 5.1 性能

| 场景 | 目标响应时间 |
|------|------------|
| OCR 识别单页扫描件 | < 10 秒 |
| 语义搜索（1000 条证据内） | < 3 秒 |
| AI 增强匹配（50 条结算项） | < 30 秒 |
| 完整性检查 | < 15 秒 |
| 自然语言解析 | < 5 秒 |

### 5.2 数据安全

- AI API 调用仅发送处理所需的文本/图片，不发送项目元数据（项目名称脱敏可选）
- 本地 Ollama 模式下所有数据不出本机
- AI 调用日志不记录原始文件内容（只记录 token 量和成功/失败状态）
- 用户可在设置中关闭任意 AI 功能

### 5.3 容错与降级

| 异常场景 | 降级行为 |
|---------|---------|
| AI API 不可用 | OCR/搜索/匹配回退到纯本地模式（无 AI 增强） |
| AI 调用超时 | 返回部分结果或跳过 AI 增强 |
| 识别置信度过低 | 提示用户手动填写，不填入低质量结果 |
| 本地模型未安装 | 功能入口置灰，提示配置说明 |

### 5.4 成本控制

- 每条 OCR 识别约消耗 500-2000 token（取决于文件复杂度）
- 每次语义搜索约消耗 50-200 token（取决于结果数量）
- AI 增强匹配每次组卷约消耗 2000-10000 token
- 预估月 token 消耗（中等使用量）：¥20-80/月（DeepSeek 计费）
- 建议：在设置中展示本月 AI 用量

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| OCR 识别率受扫描件质量影响 | 识别结果不可用 | 「AI 建议 + 人工确认」模式，低置信度不回填 |
| AI API 延迟不稳定 | 用户体验差 | 设置超时（15s），超时自动降级 |
| 敏感结算数据上传云端 | 用户顾虑数据安全 | 支持本地 Ollama，默认推荐但可选 |
| AI 匹配误判导致漏匹配 | 结算证据不全 | AI 匹配标注 `matchKind`，用户可一键移除 |
| Embedding 模型切换导致向量不兼容 | 语义搜索失效 | 记录 model 版本，切换时提示重建索引 |
