# 证据链管理系统 — 代码审查与优化方向

> 审查日期：2026-06-20
> 代码规模：前端 2,944 行 + 后端 3,022 行 + 测试 885 行 = 5,966 行业务代码
> 行数 TOP 3：App.jsx (1,444) · styles.css (1,338) · server/index.js (704) — 占源码 58%

---

## 一、架构层面

### 方向 1：App.jsx 单体组件拆分（1,444 行 → 多文件）

**现状**：一个 `App.jsx` 包含了所有页面组件、所有表单（4 种证据 + AI 设置）、所有 UI 小组件、水印生成逻辑、状态管理。这是当前最大的结构性问题。

**拆分建议**：

| 拆出目标 | 建议路径 | 包含内容 |
|---------|---------|---------|
| 页面组件 | `src/pages/` | `Dashboard.jsx`、`EvidencePage.jsx`、`PackagePage.jsx`、`SearchPage.jsx`、`SettingsPage.jsx` |
| 证据表单 | `src/forms/` | `VariationForm.jsx`、`HiddenRecordForm.jsx`、`MaterialRecordForm.jsx`、`MonthlyMeasurementForm.jsx` |
| 共享 UI 组件 | `src/components/` | `Metric.jsx`、`EvidenceList.jsx`、`FileDrop.jsx`、`StatusBadge.jsx`、`FormPanel.jsx` 等 |
| AI UI | `src/ai/` | `AiSettingsForm.jsx`、`AiRecognizeButton.jsx`、`AiResultNote.jsx`、`CompletenessBadge.jsx` |
| 工具函数 | `src/lib/` | `watermark.js`（水印生成）、`formatters.js`（`evidenceSummary`/`evidenceDetail`/`formatNumber`）、`formHelpers.js`（`applyDraftToForm`/`aiSettingsPayloadFromForm`） |

**收益**：单文件从 1,444 行降到每文件 100-300 行；组件可独立测试；新人/AI 辅助开发时更容易定位。

---

### 方向 2：server/index.js 路由拆分（704 行 → 多文件）

**现状**：所有 API 路由（证据 CRUD、结算、AI、搜索）全在一个文件里，加上内联辅助函数（`persistEvidenceFiles`、`semanticRankEvidence`、`parseSpreadsheetFile` 等），路由定义和业务逻辑混在一起。

**拆分建议**：

```
server/
├── routes/
│   ├── projects.js        # /api/projects/*
│   ├── evidence.js        # /api/projects/:id/variation-orders, hidden-records 等
│   ├── settlements.js     # /api/projects/:id/settlements/*
│   ├── search.js          # /api/search, /api/search/semantic
│   └── ai.js              # /api/ai/*
├── index.js               # 只做 app 组装 + 中间件注册（<100 行）
```

**收益**：每个路由文件 < 150 行；新增功能时不用在 700 行里找插入位置；路由可独立测试。

---

### 方向 3：状态管理升级

**现状**：所有状态通过 `useState` 在 `App` 组件管理，然后通过 props 层层传递（prop drilling）。`PackagePage` 尤其严重 — 它管理了 `pasteText`、`parsed`、`columns`、`payload`、`activeItemId`、`preview` 共 6 个 useState，同时接收 `evidence`、`sessions`、`runAction` 等 props。

**建议方向**：

| 方案 | 改动量 | 适用场景 |
|------|--------|---------|
| **A（轻量）**：React Context 拆分为 `ProjectContext`（projectId + projects + overview + evidence + sessions）和 `AiContext`（aiStatus + aiFeatures） | 小 | 仅消除顶层 props 传递 |
| **B（中量）**：引入 Zustand，store 按域拆分：`useProjectStore`、`useEvidenceStore`、`useSettlementStore`、`useAiStore` | 中 | 状态逻辑从组件中抽离，可独立测试 |

**收益**：消除 props 传递链路（目前 `PackagePage` 接收 6 个 props，深度嵌套传递到 5 层子组件）；状态更新逻辑从组件中抽离。

## 二、性能层面

### 方向 4：语义搜索的 N+1 Embedding 调用

**现状**：`semanticRankEvidence`（`server/index.js:599`）对每条 evidence 逐个调用 `provider.embed(text)` 做向量化。200 条证据 = 200 次 API 调用。

```javascript
// 当前做法（N+1）：
for (const record of evidence) {
  const vector = await provider.embed(text);  // 每条一次 API 调用
  scored.push({ ...record, semanticScore: cosineSimilarity(queryVector, vector) });
}
```

**建议**：
- 使用 `evidence_embeddings` 表（`db.js` 中已建好但未被 `semanticRankEvidence` 使用）。新建/修改证据时预计算向量存入 DB，搜索时只对查询文本做 1 次 embedding，然后从 DB 批量取向量做余弦相似度
- 降级方案：首次搜索时对结果集做批量 embedding（一次 API 调用传多个 input，如果 provider 支持）

**收益**：搜索从 N+1 次 API 调用降为 1-2 次；搜索响应时间从 10-30 秒降到 < 2 秒。

---

### 方向 5：getSettlementPayload 重复调用

**现状**：一次请求链路中 `getSettlementPayload` 被多次调用（如 `buildPreview` 调用一次，返回的 handler 又调用一次），每次都会重新查询 DB 并做 `listFilesForEvidence` + `withFiles` join 操作。

**建议**：
- 在请求级别缓存 `getSettlementPayload` 的结果（用 `req.locals` 或简单的 Map，请求结束后释放）
- 或者将其改为接收已构建好的 payload，由调用方传参而非内部重建

---

### 方向 6：CSS 文件体积

**现状**：`styles.css` 共 1,338 行，全部在一个全局文件中。没有使用 CSS Modules 或 scoped styles。

**建议**：
- 迁移到 CSS Modules（Vite 原生支持，无需额外配置）：`.field` → `FormPanel.module.css`
- 或者引入 Tailwind CSS（RebarVis Pro 项目已用过，配置熟悉）
- CSS 自定义属性（`--accent`、`--panel` 等）已用得很好，保持这个模式

---

## 三、数据流与一致性

### 方向 7：证据类型常量化

**现状**：「variation」「hidden」「material」「monthly」这四个字符串散落在约 30 处代码中：`App.jsx`、`server/index.js`、`matching.js`、`checker.js`、`ocr.js`、`repository.js`。如果新增证据类型（如 P2 的「联系函」），需要改 6-8 个文件。

**建议**：
- 创建 `shared/constants.js`（或 `server/lib/evidenceTypes.js`），统一定义：
  ```javascript
  export const EVIDENCE_TYPES = ['variation', 'hidden', 'material', 'monthly'];
  export const EVIDENCE_LABELS = { variation: '签证变更单', ... };
  export const EVIDENCE_TYPES_FOR_MATCHING = new Set([...]);
  ```
- 所有模块从这里引用，前端通过 API 或构建时共享

---

### 方向 8：前后端证据类型和字段的重复定义

**现状**：前端 `App.jsx` 的 `evidenceLabels` 和后端 `checker.js` 的 `TYPE_LABELS` 以及 `ocr.js` 的 `FIELD_SCHEMAS` 各自维护了一份字段定义。前端表单和后端 API 校验规则也互相独立。

**建议**：
- 将 `FIELD_SCHEMAS`（已在 `ocr.js` 定义，包含 fields/required/numbers/booleans）提升为唯一的字段定义源
- 前端通过 API 获取字段 schema（`GET /api/evidence/schemas`），动态渲染表单和校验
- 或者前后端共享一个 JSON/YAML schema 文件（`shared/evidenceSchemas.json`）

---

### 方向 9：数据库 `payload_json` 字段的读写一致性

**现状**：
- `server/index.js` 中创建 evidence 时手工构造 payload 对象（每个证据类型 6-8 个字段）
- `App.jsx` 的 `evidenceSummary` 函数手工解 `payload.materialName`、`payload.spec` 等字段
- `matching.js` 也用 `evidence.payload.materialName` 等硬编码访问

**建议**：
- 为每个证据类型定义 payload 的 TypeScript 接口或 JSDoc `@typedef` 注释（不改构建，纯文档约束）
- 或者用 Zod schema 在 `createEvidence` 时校验 payload，确保写入的数据结构一致
- 前端 `evidenceSummary`/`evidenceDetail` 改成按证据类型的策略模式（类似现有的 if/else，但抽成独立函数）

---

## 四、AI 模块专项

### 方向 10：AI 错误重试和降级策略

**现状**：`CompatibleAiProvider.request()` 有一次超时控制（`AbortController`），但没有重试逻辑。网络抖动或 provider 限流会导致用户操作直接失败。

**建议**：
- 在 provider 层增加指数退避重试（1 次重试，delay 1s）
- OCR 识别失败时，返回部分识别结果（已识别的字段）+ 明确的错误提示，而非整个请求失败
- `semanticRankEvidence` 目前 AI 调用失败时直接 throw，建议降级为关键词搜索结果 + `semanticFallback: true` 标记

---

### 方向 11：AI 设置存储的安全加固

**现状**：API Key 明文存储在 SQLite 的 `ai_settings` 表中（`api_key TEXT`）。虽然有 `publicAiConfig` 做脱敏返回，但数据库文件本身没有加密。

**建议**：
- 使用 Node.js `crypto` 模块对 API Key 做对称加密存储（密钥来自环境变量 `AI_ENCRYPTION_KEY` 或机器指纹）
- 如果用户不配环境变量，至少提示 "API Key 以明文存储在本地数据库" 的风险

---

### 方向 12：AI 匹配的串行调用瓶颈

**现状**：`buildAiMatchCandidates`（`matcher.js`）对每个被规则引擎拒绝的配对都串行调用 `provider.judgeEvidenceMatch`。如果 50 条结算项 × 100 条证据 = 5,000 个 rejected，即使是限制 50 次也是串行 50 次。

**建议**：
- 改为并发批处理：用 `Promise.allSettled` 并发 N 个（如 5 个并发），有速率限制
- 预先过滤：按时间范围快速过滤明显不相关的（证据日期和结算项时段差 > 6 个月的直接跳过）

### 方向 14：TypeScript 或 JSDoc 类型标注

**现状**：项目是纯 JavaScript，没有任何类型标注。`evidence.payload` 在 8+ 个地方以不同方式被访问，IDE 无法提供自动补全或类型检查。

**建议**：
- 渐进式引入：在不改构建的前提下，给核心数据结构加 JSDoc `@typedef` 注释
  ```javascript
  /**
   * @typedef {{
   *   id: string, type: string, title: string, location: string,
   *   evidenceDate: string, amount: number|null,
   *   payload: Record<string, any>, files: FileRecord[]
   * }} EvidenceRecord
   */
  ```
- 长期：考虑迁移到 TypeScript（`vite` 原生支持 `.tsx`，增量迁移成本低）

## 六、用户体验层面

### 方向 15：表单操作体验

**现状问题**：
- v1.0 暂不支持编辑和删除，但 UX 上没有任何反馈入口
- 保存成功只是 toast 一闪而过（2.6 秒），没有持久的状态指示
- 表单提交后 `event.currentTarget.reset()` 清空，填错想修正需要重新填

**建议**：
- 证据列表项点击后展示详情 + 预留"编辑"/"删除"按钮位置（置灰标注"即将支持"）
- Toast 后增加一个「查看刚保存的记录」链接，点击滚动到列表中的新记录
- 编辑功能（v1.1 规划中）优先实现

---

### 方向 16：加载和空状态的细化

**现状**：多处用 `EmptyText` 组件展示通用文案。导入结算项解析失败时没有区分不同失败原因。

**建议**：
- 证据列表为空时根据当前 Tab 显示引导文案（如"还没有签证变更记录。上传第一份签证单，开始建立证据链。"）
- 导入结算项增加失败原因分类反馈（格式错误 / 表头缺失 / 仅识别到 N 行等）
- Dashboard 无项目时引导创建第一个项目（已有 `ProjectRequiredPage`，保持）

## 七、工程与安全

### 方向 17：依赖版本管理

**建议**：
- 运行 `npm audit` 检查已知漏洞
- 在 AGENTS.md 中注明 `npm ci`（而非 `npm install`）用于 CI/部署
- `multer` 增加上传文件大小限制：`limits: { fileSize: 50 * 1024 * 1024 }`

---

### 方向 18：文件上传清理

**现状**：上传文件写入 `TMP_DIR`，OCR 路由在 finally 中清理，材料进场等路由校验失败时调 `validationError` → `discardUploads` 清理。逻辑分散。

**建议**：
- 统一为 multer 中间件 + 请求结束时的统一清理逻辑（Express middleware after-response hook）
- `persistUpload` 成功移动文件后，确认 TMP_DIR 中无残留
